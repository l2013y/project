userid、appid和openid的解释

userId 是指系统里的用户ID，存在于数据库表中，例如：

id username
001 user@example.com

其中的001就是用户ID，代表的是系统中的某个账号，只要这个系统用户不被删除重建，userId 一般不会变。

appId 是微信给每个小程序分配的ID，目前这个项目所用的 wxe612…. 就是appid，相当于微信小程序的身份证。

openId 是微信给”某个微信用户在某个小程序下”生成的唯一 ID。代表某个微信用户在这个小程序中的身份，更改微信号、换小程序appId会导致openId的改变。

AI 代码质量检查维度

让 AI 写代码时，不能只看“能不能跑”，还要从正确性、安全性、性能、可维护性等角度检查。下面这张表可以当作一个简单的自查清单：

维度 主要看什么 常见坏味道 更好的做法
正确性 代码是否按预期运行 边界条件没处理、空指针 单元测试覆盖、异常处理完善
安全性 是否防止注入、泄露、越权 SQL 拼接、密码明文、无权限校验 参数化查询、敏感信息加密、权限控制
性能 响应是否够快，资源占用是否合理 循环查数据库、大对象不释放 批量操作、缓存、懒加载
可维护性 后续是否好改、好扩展 改一处要改十处、逻辑分散 高内聚低耦合、配置与代码分离
可测试性 是否方便写测试验证 逻辑全写在 UI 里、强依赖外部环境 业务逻辑抽离、依赖注入
兼容性 不同环境下是否都能运行 硬编码路径、默认某个系统环境 相对路径、环境变量、配置外置

表格内术语简单解释：

  • 边界条件：正常情况之外的特殊情况，比如输入为空、数字特别大、列表没有数据。
  • 空指针：代码以为某个对象一定有值，实际却是空的，继续使用就会报错。
  • SQL 拼接：把用户输入直接拼到 SQL 语句里，容易被恶意输入利用。
  • 参数化查询：把用户输入当作参数传给数据库，而不是直接拼字符串，更安全。
  • 密码明文：密码没有加密或哈希处理，直接以原始文字保存或传输。
  • 权限校验：检查当前用户有没有资格执行某个操作，比如删除、查看、修改数据。
  • 缓存：把常用数据临时存起来，避免每次都重新查数据库或重新计算。
  • 懒加载：需要用到时才加载数据，不是一开始就全部加载。
  • 高内聚低耦合：相关逻辑尽量放在一起，不相关的模块之间少互相依赖，后续更好改。
  • 依赖注入：把外部依赖传进来，而不是在代码内部写死，方便测试和替换。
  • 硬编码路径：把具体路径直接写死在代码里，比如某台电脑上的固定目录，换环境就容易失效。
  • 环境变量：把端口、密钥、数据库地址等配置放到运行环境里，不直接写死在代码中。

简单来说，AI 生成代码后可以先问自己一句：它只是“看起来能跑”，还是在异常情况、安全风险、后续维护上也站得住？

今天下午先是用deepseek帮我生成了一份语音转文字的提示测，之后复制提示词让claude帮我生成了语音转文字的基础网页代码，先是生成了index.html和server.js,之后在文件夹中打开终端 npm install ws 安装了依赖，之后在cscode的终端中使用node server.js启动了服务。

打开网页之后发现无法实现语音转文字的这个功能，因为一开始只给了Claude百度模型的APP ID，之后经过一番调试之后，在百度智能云创建了API key，Secret key，access_token，并把这些一股脑都给claude之后再次调试就成功了！

失败原因总结

  • 阶段 1

    • 问题：火山引擎401错误
    • 原因：豆包语音识别需要复杂的V3签名认证，不是简单在Header里放APP_ID就行，百度是OAuth，火山是HMAC签名，机制完全不同
    • 解决方式：放弃火山，改用百度

  • 阶段 2

    • 问题：百度3300错误 url param cuid error
    • 原因：参数位置放错了：token 应该放body里，不是url参数；cuid 格式有限制，不能带下划线
    • 解决方式：修正参数位置，cuid改成纯数字

  • 阶段 3

    • 问题：识别结果为空 result:[“”]
    • 原因：音频采样率转换有bug：浏览器48kHz采样 → 代码里降采样到16kHz时算法有问题，生成的音频数据百度无法识别
    • 解决方式：直接让AudioContext用16kHz采集，跳过降采样步骤

  • 阶段 4

    • 问题：重复识别、重复文字
    • 原因：点击停止后：按钮没禁用，可以重复点；2秒超时逻辑又会触发stop导致后端收到N次stop，识别N次
    • 解决方式：加 disabled 锁，后端加防重复处理

博客方面的优化

  • 1. 响应式布局问题

    问题：手机端侧边栏头像、目录卡挡在文章内容前面，无法阅读和点击。

    解决：修改 flex 布局，移动端改为 flex-direction: column；文章内容 order: 1（上面），侧边栏 order: 2（下面）；移动端设置 position: static 清除定位，避免覆盖内容。

  • 2. 移动端所有按钮点击失效

    问题：手机端进入文章页面，除了返回顶部按钮外其他都点不了，有密码的文章也输入不了。

    根因：遮罩层 .toc-overlay 在移动端默认是透明的（opacity: 0）但仍存在于页面上，看不见但覆盖了整个页面，拦截所有点击事件。

    解决：添加 visibility: hidden，未激活时完全不拦截点击；打开目录时才设置 visibility: visible。

  • 3. PC 端显示移动端悬浮目录按钮

    问题：PC 端左下角也出现了移动端才有的目录按钮。

    解决：双重保险——CSS 加 display: none !important + JavaScript 窗口宽度判断，只有宽度 ≤ 991px 才显示按钮。

  • 4. 点击目录出现黑框

    问题：PC 端点击目录链接后，项目上留下一个黑色聚焦边框。

    解决：全局加 *:focus { outline: none !important; }，移除所有元素的聚焦样式。

  • 5. 目录按钮与返回顶部高度不一致

    解决：统一两个按钮的尺寸——50px × 50px，bottom: 30px，font-size: 24px。现在左下角目录按钮和右下角返回顶部按钮完全对称对齐。

最终效果：PC 端侧边栏 + 粘性目录，无多余悬浮按钮；移动端左右两个对称悬浮按钮，点击目录弹出侧边面板；加密文章解密后目录正常生成，密码输入可交互。

2026.5.26 登录问题

1. PC 小程序登录提示”网络异常”

小程序 PC 端输入账号密码后提示”网络异常，请稍后再试”，控制台报 POST https://xxxx.ngrok-free.dev/app/... 502 Bad Gateway。

原因：一开始前端请求的是 ngrok 地址 https://xxxx.ngrok-free.dev/app/...，但是本地后端端口当时没有正常监听，所以 ngrok 转发不到后端，返回 502。

解决：确认后端启动后，本地接口 curl -i http://127.0.0.1:后端端口/app/... 返回 {"success":false,"message":"Request method 'GET' not supported"}（说明后端已跑通，只是 GET 不支持）。之后用 POST 测试 curl -i -X POST "http://127.0.0.1:后端端口/app/..." -d "username=测试账号&password=测试密码"，ngrok 地址也测试成功，网络链路恢复。

2. MySQL 提示 No database selected

执行 SQL 时提示 ERROR 1046 (3D000): No database selected。原因是进入 MySQL 后还没有选择数据库。解决：先执行 USE 数据库名;。

3. curl 在 MySQL 里执行导致一直换行

在 MySQL 里输入 curl 后出现 -> 不断换行。原因是 curl 是 Windows 命令行命令，不是 SQL，当时还停留在 MySQL 命令行里，MySQL 把它当成未完成 SQL 处理。解决：先 exit; 退出 MySQL，回到命令行再执行 curl。

4. curl 登录成功，但小程序显示”账号不存在”

curl 返回 "success": true，但小程序 Network 返回 {success: false, code: 2000, message: "账号不存在"}。

原因：curl 默认用表单格式 application/x-www-form-urlencoded，而小程序请求可能按 JSON 方式提交。后端登录方法 public R login(@ParameterObject LoginDto loginDto, HttpServletRequest request) 不是 @RequestBody，更适合接收表单参数。

解决：修改 .../pages/loginPage/index.vue，登录请求增加 headers: { 'content-type': 'application/x-www-form-urlencoded' }；同时修改 .../src/main.js，避免请求拦截器覆盖单个请求自己的 headers。

5. 本地调试不再使用 ngrok

因为不需要上传体验版，只做本地调试。把小程序接口地址从 https://xxxx.ngrok-free.dev/app 改成 http://127.0.0.1:后端端口/app，修改 .../config/dev.env.js 和 .../config/prod.env.js。PC 微信开发者工具模拟器可以正常访问。

6. 手机真机调试不能用 127.0.0.1

PC 模拟器正常，但手机真机提示网络异常。原因：电脑上 127.0.0.1 指电脑自己，手机上 127.0.0.1 指手机自己，手机上没有 Java 后端。

解决：查到电脑局域网 IPv4 为 局域网IP，把接口地址改成 http://局域网IP:后端端口/app，修改 .../dev.env.js 和 .../prod.env.js 并重新构建。手机浏览器访问 http://局域网IP:后端端口/app/... 能返回后端 JSON，说明链路成功。

7. 真机调试提示 调试通道端口 局域网连接失败

微信开发者工具提示”连接局域网 局域网IP:调试通道端口 失败，已切回广域网模式”。调试通道端口 是微信开发者工具自己的真机调试通道端口，不是项目后端端口（项目后端端口）。手机浏览器能访问 后端端口，说明项目接口本身没问题。继续使用真机调试即可，后续可检查 Windows 防火墙。

8. 临时关闭静默登录

为了方便手机端直接测试账号密码登录，.../App.vue 把启动时的 wx.login() → getTokenByCode 静默登录流程去掉，改成启动直接跳转登录页：

onLaunch: function (option) {
wx.navigateTo({ url : ‘/pages/…/main’ })
}

9. 手机端最终登录成功

手机端使用另一个未绑定账号成功登录。下面这条链路已经打通：

手机小程序 → http://局域网IP:后端端口/app → 本地 Java 后端 → 本地 MySQL 数据库

本地真机调试环境已经成功跑通。

本地调试配置备忘： 后端端口，接口上下文 /app，本地数据库 本地数据库名，Redis 默认端口。小程序接口地址使用 http://局域网IP:后端端口/app（同时兼容 PC 模拟器和手机真机）。

常见错误判断：

  • 网络异常，请稍后重试 —— 优先检查：后端端口是否启动、手机是否和电脑在同一 Wi-Fi、小程序 BASE_URL 是否是 局域网IP:后端端口、手机浏览器能否访问 http://局域网IP:后端端口/app/...。
  • 账号不存在 —— 说明 username 没查到，检查是否输错账号、数据库 sys_user.username 是否存在、请求参数是否正确传到后端。
  • 账号密码错误 —— 账号存在，但密码不对。
  • 账号已经被其他微信绑定 —— 账号已绑定别的 open_id，可换账号，或本地开发时清绑定：USE 本地数据库名; DELETE FROM sys_user_wx WHERE user_id = 用户ID; UPDATE sys_user SET bind_wx = 0 WHERE id = 用户ID;

2026.5.27 小程序 AI 语音转文字 — 从踩坑到上线

1. 功能定位：第一版只做巡店总结

巡检表单里有很多字段：合格/不合格、打分、单选、情况说明、签字、巡店总结。起初容易被”AI 控制整张表单”的思路带偏，但逐个分析后发现：

  • 合格/不合格 → 点一下就行，语音反而增加误判风险
  • 打分、单选 → 需要精确解析，第一版实现成本高
  • 签字 → 不适合语音
  • 巡店总结 → 长文本，手动输入成本高，语音收益最大

收获：第一版功能要克制，选收益最高、风险最低的字段切入，不要一上来就追求”全自动”。

2. 入口问题：找到真正的页面

小程序底部导航是”任务、报告、问题、汇总、我的”，没有”巡检”这个入口。实际流程是：任务列表 → 点某个任务的”巡检”按钮 → 进入巡检详情页。

收获：页面名称和用户操作入口不一定一一对应，要看代码里的跳转逻辑确认真实页面。

3. 组件 vs 页面内实现

一开始想做通用语音组件给其他页面复用，但 mpvue 的组件编译和开发者工具缓存叠加后，容易出现”找不到组件 WXML 文件”的报错。后来把语音按钮直接写在页面里，功能快速跑通。

收获：复用是方向，但如果框架编译不够稳，先跑通再抽象。源码、构建产物、开发者工具缓存三者可能不一致，清缓存是基本功。

4. 百度云切腾讯云

认证方式完全不同：百度云用 OAuth token（需缓存+续期），腾讯云用 TC3-HMAC-SHA256 签名（每次请求动态计算）。换云服务不只是换 URL。

5. Network Error 排查

网页端全站报 Network Error，后端服务正常。根因是前端 axios 配置 withCredentials: true，后端 CORS 配置 allowCredentials: false，浏览器检测到不匹配直接拦截。

收获：后端无日志 + 前端 Network Error → 优先查 CORS credentials。

6. 微信 API 异步陷阱

wx.showModal 弹提示后代码继续往下执行，弹窗和录音”一起发生”，弹窗被覆盖。弹窗后需要 return，在回调里启动录音。

收获：微信 API 几乎是全异步的，弹窗不会阻塞代码。

7. 权限按钮的坑

加了个删除按钮，按钮不显示。因为用 showBtnFn 做了权限校验，但数据库权限表里没这个 key。新按钮如果配了权限，先确认权限数据有没有。

8. 语音录入的目标选择

每个检查项都有自己的”情况说明”文本框，全局就一个语音按钮，录给谁？

当前设计：默认录入巡店总结；如果用户先点击打开了某个”情况说明”文本框，则优先录入那个。首次使用弹提示：”语音内容默认录入巡店总结，如需写入情况说明，请先点击对应输入框”。

收获：UI 简洁比功能强大更重要，一个按钮 + 上下文判断，比到处放按钮体验好。

2026.5.28 小程序 AI 语音与巡检保存问题整理

1. AI 语音功能重新收敛

昨天的方向是让 AI 语音尽量智能地匹配检查项，甚至自动判断合格/不合格。今天重新梳理后发现，当前最稳定、最适合上线的版本还是“语音转文字录入文本框”：默认录入巡店总结；如果用户先点击某个情况说明输入框，就把语音内容写入对应输入框。

收获：第一版不要急着做“AI 自动填整张表”。语音识别先解决长文本输入效率，智能判断可以后面再加，而且最好做成“AI 分析结果预览 + 用户确认填入”，避免误填。

2. 录入目标和高亮逻辑

点击 AI 语音按钮时，文本框可能会失焦，如果只依赖焦点状态，语音结果就会错误地回到巡店总结。处理方式是用 voiceTarget 记录最后选中的录入目标，不再依赖真实焦点；同时给当前目标加蓝色高亮，让用户知道语音会写到哪里。

首次提示也改得更明确：默认录入巡店总结；点击输入框高亮后，可将语音内容录入到对应位置。

3. 模拟和自动判断先撤回

中途尝试过加“模拟”按钮，用手动输入文本模拟语音内容，并用关键词规则自动选择合格/不合格，比如“故障、异常、不正常”判为不合格，“正常、合格、没问题”判为合格。

但关键词规则不是真智能，复杂表达很容易误判，比如“不是不合格”“没有异常但有轻微噪音”。最后决定删除模拟按钮、模拟弹窗、自动匹配合格/不合格逻辑，同时删除 index.vue.bak、index.vue.work 这类备份文件，避免旧提示和旧代码干扰判断。

4. 微信开发者工具与导入目录问题

开发者工具渲染层出现过 getGlobalStorage fail: too eayly，试过临时关闭静默登录后仍然存在，但功能不受影响，暂时判断是微信开发者工具初始化层面的提示，不继续改业务代码。

另外，不能直接导入 dist 目录，否则容易出现网络和配置异常。这个项目应该导入 shoptour-frontend 根目录，因为根目录的 project.config.json 已经配置了 miniprogramRoot: "dist/"。

5. 巡检保存失败问题

点击“完成收起”时曾出现“操作失败！null”。后端排查后发现，保存报告时有些字段可能为空：reportItem.getScore()、redline、passline 都可能触发空值问题。

处理方式：分数为空时按 0 汇总；redline 改成安全判断；passline 为空时按 0 处理；提交报告时，空分数也走空白提交逻辑。

6. itemId 接收不到问题

保存巡检时还出现过“巡店项目不存在：null”和 Required request parameter 'itemId' is not present。原因是前端用 POST 提交时，参数在 body 里，后端一开始没有稳定接收到。

处理方式：前端保存时把 itemId 转成字符串；后端 /setReport 改成兼容 POST body 和普通参数。这样保存链路更稳，不会因为参数位置不同导致后端拿不到巡店项目。

当前结论： AI 语音第一版只保留稳定的语音转文字录入；巡检保存问题重点处理空值和参数接收。后续如果继续做智能化，应该让大模型结合“语音文字 + 当前检查项标题 + 表单结构”返回结构化结果，再让用户确认后填入。

2026.5.29 自动AI录音与保存体验优化

1. 点击”完成收起”时出现网络超时提示

确认后端可以正常访问后，把前端请求超时时间从 10000ms 调到 30000ms，保存可以成功，但接口响应仍然偏慢，后续还可以继续优化后端保存性能。

2. 选择完全不合格时自动调用 AI 语音

实际业务中，部分合格或扣分项也需要说明原因，所以处理成：只要选择非合格或低于满分项，就自动展开对应的”情况说明”，并把 AI 语音录入目标切换到该说明框。

3. 增加”自动AI”开关

为了避免自动录音打扰用户，关闭时只展开说明框，不自动录音；开启时选择非合格项会自动开始录音。开关放在右下角 AI 语音按钮上方的小三角里，点击后展开或收起，比单独放按钮更清爽，也不容易误触。

4. 自动AI 首次提示和按钮点击修正

自动AI 第一次触发时，提示曾经到停止录音时才弹出。处理后，自动触发和手动触发统一走首次提示逻辑；录音过程中点击按钮只停止录音，不再弹提示。小三角和 AI 语音按钮功能反了的问题，也通过 data-action 分发点击行为解决。

5. 延长识别失败提示时间

“未识别到语音”的提示一闪而过，用户来不及看清。现在把识别失败和上传失败的提示延长到 3000ms，同时避免和”识别中”的 loading 重叠。

6. 增加自动录音状态提示

自动录音开始时，用户不一定知道已经开始。现在增加了右下角气泡提示”已开始录音，请说明原因”，带淡入淡出效果，自动录音状态更清楚。

当前结论： 自动AI 开关让语音录入更灵活，用户可自主选择是否自动触发；按钮交互和提示体验已打磨到位，非合格项自动联动展开说明框的逻辑也已跑通。

2026.6.1第一版所碰到的问题

在让claude帮忙增加AI语音转文字的功能时没有对其进行限制，更改了许多的项目配置文件和登录逻辑，主要的原因是当时没有这些安全意识，之后的任务中首要就是让AI不要随意更改项目本身的配置和登录逻辑。另外的问题就是编译代码一定要保证其可读性以及之后的维护简易性，简单来说就是能写全局插件或者复用组件优先。

2026.6.2~6.4 第二版项目

巡店系统 AI 语音功能问题总结。

1. 小程序运行的是 dist，不是 src

  • 问题：改了 src 代码，但微信开发者工具没有变化。
  • 原因：mpvue 项目实际运行的是 dist/ 目录。
  • 解决：只改 src，然后运行 npm run dev 重新生成 dist。

2. npm run build 会生成生产环境包

  • 问题：本地调试时接口地址突然变成线上地址，甚至出现 Cannot read property 'call' of undefined。
  • 原因：npm run build 会按生产配置生成 dist。
  • 解决：本地调试使用 npm run dev，不要用 npm run build 调试小程序。

3. 真机调试不能用 localhost

  • 问题：模拟器能登录，真机输入账号密码没反应。
  • 原因：真机里的 localhost 指的是手机自己，不是电脑。
  • 解决：真机调试要用 http://电脑局域网IP:后端端口/app，或直接使用公司测试环境地址。

4. 登录状态缓存会掩盖问题

  • 问题：配置改错了，但页面好像还能打开。
  • 原因：小程序缓存了 token、用户信息、权限信息。
  • 解决：排查登录问题时清缓存，重新登录。

5. AppID / AppSecret 一定要恢复

  • 问题：本地真机调试临时用了个人 AppID，提交前容易忘记恢复。
  • 影响：会导致同事环境、测试环境微信登录异常。
  • 解决：提交前检查各项配置文件，确认没有个人 AppID / AppSecret。

6. AI 密钥不能放前端

  • 问题：腾讯云 SecretId / SecretKey 如果写前端会泄露。
  • 解决：密钥只放后端，推荐通过环境变量 TENCENT_ASR_SECRET_ID、TENCENT_ASR_SECRET_KEY 配置，再由后端读取。

7. 腾讯云 ASR 不是大模型

  • 问题：容易误以为接入的是“大模型”。
  • 实际：本次使用的是腾讯云 ASR 语音识别，SDK 包为 com.tencentcloudapi.asr.v20190614，作用只是语音转文字。

8. 小程序老项目不适合贸然挂复杂组件

  • 问题：尝试挂 Vue 语音组件后，mpvue 页面容易异常。
  • 解决：改成页面级普通按钮，直接在巡检详情页接入语音功能。

9. “点击不合格自动录音”风险较高

  • 问题：自动录音涉及权限、误触、重复点击、录音状态管理。
  • 解决：改成更稳的方案：点击不合格后，自动展开情况说明，记录当前语音填入目标；之后由用户手动点击 AI 语音，识别结果再填入当前情况说明。

10. 语音默认填入巡店总结

  • 问题：没有选中具体说明框时，识别结果需要有默认去处。
  • 解决：默认填入巡店总结；如果当前有非合格项目标，则填入对应情况说明。

11. “全部合格 / 全部不合格”要避免误触发

  • 问题：如果只说“合格”就全选，容易误操作。
  • 解决：只有识别到 全部、全都、都、所有、全选 这类关键词时才批量处理，例如“全部合格”“都不合格”。

12. 巡检项是懒加载的

  • 问题：未展开的检查项没有子项数据，无法直接全选。
  • 解决：执行“全部合格/不合格”前，先加载所有未展开明细。

13. 不同题型赋值方式不同

  • 问题：只改界面选中状态，保存时可能不生效。
  • 解决：按题型分别处理：对错型改 passName 和 score；打分型改 score；选择型改 ischeck、options[].ischeck 和 score。

14. 情况说明有 200 字限制

  • 问题：AI 识别结果过长会超过文本框限制。
  • 解决：填入情况说明时截断到 200 字。

15. 后端要分 app 和 admin 两套接口

  • 问题：小程序和网页管理端不是同一个后端模块。
  • 解决：两个模块都要加 Controller 和腾讯云 SDK 依赖。

16. 后台管理端适合做全局组件

  • 问题：每个表单页面单独改成本太高。
  • 解决：在 App.vue 全局挂 AI 语音组件，自动识别当前弹窗/抽屉表单。

17. 登录页不要显示 AI 按钮

  • 问题：全局组件会出现在登录页。
  • 解决：给组件加路由判断，例如 <ai-voice-to-text v-if="$route.path !== '/login'" />。

18. 提交前必须检查配置文件

  • 避免提交：本地 IP、个人 AppID、个人 AppSecret、构建产物、开发工具私有配置、无关锁文件变化。

19. 远程旧分支可用 –force-with-lease 覆盖

  • 问题：远程 dev2026 有旧的大改动提交，普通 push 会被拒绝。
  • 解决：确认需要用本地分支覆盖远程旧分支时，可以使用 git push --force-with-lease origin 本地分支:dev2026。它比 --force 更安全，但执行前仍然要确认远程旧提交确实可以被覆盖。

20. git fetch 不会改本地代码

  • 问题：担心 git fetch 会拉代码覆盖本地。
  • 实际：git fetch 只更新远程分支记录，不改工作区；会改代码的是 git pull。

总结： 本次问题核心不是 AI 语音本身，而是老项目多端联调：小程序构建产物容易混乱，真机和模拟器地址不同，微信登录依赖 AppID/AppSecret，前端、后台、后端三端都要同步，配置文件提交前必须仔细检查。

最终经验：老项目加新功能，要最小侵入、少改配置，先保证登录和环境稳定，再逐步增强功能。

2026.6.5 对于意见反馈的功能的方案规划

下一步就着手于对小程序端的“我的”页面添加意见反馈，意见反馈中包括问题反馈和bug反馈，然后点击这两个入口能跳转到问题填写页面，填写页面的话包括文本、图片上传和联系方式

2026.6.8 对于小程序和后台管理端的意见反馈的初步开发

这次任务是给小程序新增“意见反馈”功能，并且在后台管理端提供查看反馈记录的页面。整体功能链路：小程序用户进入“我的”页面，点击意见反馈，选择反馈类型，填写反馈内容、上传图片、填写联系方式后提交。后端接收数据后保存到数据库，同时记录提交人的用户信息。后台管理端再提供一个反馈管理页面，用来查看用户提交的反馈内容。这次功能表面上看只是一个简单的表单提交，但实际涉及了小程序前端、后端接口、数据库表、后台管理端页面、菜单权限、环境配置等多个环节。

1. 后端为什么要按分层结构实现

一开始需要为意见反馈新增后端保存接口。这个功能虽然简单，但没有直接在接口里写数据库操作，而是按照项目原有结构拆分为实体类、数据库操作层、业务服务层和接口层。
这样做的原因是：

  • 保持和项目原有代码风格一致；
  • 后续后台管理端查询反馈时可以复用同一套业务模块；
  • 如果以后要增加处理状态、回复、删除等功能，扩展会更方便；
  • 避免把数据库操作直接写在接口层，降低维护成本。
    虽然文件数量比“只写一个接口”更多，但这是符合项目规范的最小完整实现。

2. 数据库表的设计

意见反馈需要单独建表保存用户提交的数据。
表里主要保存这些信息：

  • 反馈类型；
  • 反馈内容；
  • 图片地址；
  • 用户填写的联系方式；
  • 提交用户 ID；
  • 提交用户名称；
  • 创建时间。
    其中，提交用户 ID 和提交用户名称不是前端传过来的，而是后端从当前登录用户中获取的。
    这样即使用户在联系方式里填错手机号，系统也仍然能知道是谁提交的反馈。
    联系方式字段的作用更多是方便后台人员后续联系用户，而不是用来判断用户身份。

3. 小程序端提交逻辑

小程序端主要做了三个页面相关的工作：

  • 在“我的”页面增加意见反馈入口；
  • 新增反馈类型选择页；
  • 新增反馈填写页。
    填写页负责收集用户输入的反馈内容、图片和联系方式，然后调用后端保存接口。
    最开始提交成功后会立即返回上一页，导致成功提示很容易一闪而过。请求异常时也没有明确提示，用户只会感觉页面加载了一下但不知道发生了什么。
    后来对提交体验做了优化：
  • 提交成功后先显示成功提示；
  • 延迟一小段时间后再返回上一页；
  • 接口返回失败时显示具体失败原因；
  • 网络或后端异常时显示统一的失败提示。
    这个改动范围很小，但明显提升了用户体验，也方便调试问题。

4. 小程序提交接口 404 问题

调试时遇到过一个问题：小程序点击提交后提示失败，控制台显示接口 404。
一开始容易误以为是接口地址写错，或者前端请求参数有问题。后来排查发现，源码里接口已经写好了，但当前运行中的后端服务并没有加载到新增的接口类。
也就是说，代码存在，但运行的服务还是旧的编译结果。
解决方式是重新构建并重启后端服务。重启后接口正常生效，小程序提交成功，数据库也能看到新增记录。
这个问题说明：后端新增接口后，不仅要看源码有没有，还要确认运行中的服务是否已经加载了最新代码。

5. 反馈数据入库验证

提交成功后，通过数据库确认新增反馈记录是否存在。
验证时重点看几个字段：

  • 反馈类型是否正确；
  • 反馈内容是否正确；
  • 图片地址是否保存；
  • 联系方式是否保存；
  • 提交用户是否自动记录；
  • 创建时间是否正确。
    确认这些字段都正常后，说明小程序到后端再到数据库的链路已经跑通。

6. 维护发版 SQL

因为这个功能新增了数据库表，所以不能只提交代码，还需要维护数据库变更 SQL。
本地开发时数据库已经建好了，但测试环境、正式环境不一定有这张表。后续发版时，需要有人执行建表 SQL，否则代码上线后接口会因为找不到表而报错。
因此单独整理了一个变更 SQL 文件，并在数据库变更说明文档里记录这次改动。
这一步主要是为了保证后续发版时数据库和代码同步。

7. 后台管理端查看反馈

小程序能提交反馈后，数据一开始只能直接查数据库查看。为了让后台人员可以直接查看反馈记录，又新增了后台管理端页面。
第一版后台页面只做只读功能，包括：

  • 反馈列表；
  • 按反馈类型筛选；
  • 按反馈用户搜索；
  • 按联系方式搜索；
  • 分页；
  • 查看详情；
  • 图片预览。
    没有做删除、回复、处理状态等功能。
    这样做是为了保持最小范围改动，不修改现有表结构，也避免一次性扩大功能范围。

8. 后台菜单应该放在哪里

新增后台页面时，需要考虑菜单放在哪个模块。
意见反馈不属于巡店任务、报告或问题处理流程，更偏向系统运营和用户反馈管理。并且数据库表也属于系统类表。
所以最终把菜单放在系统管理模块下，和用户管理、角色管理等菜单同级，并放在菜单底部。
这样更符合功能定位，也不会影响原有巡店业务模块。

9. 后台验证码加载失败问题

调试后台管理端时，遇到登录页验证码加载不出来的问题。
排查后发现，本地前端请求的后台地址仍然指向远程环境，而远程验证码接口当时返回异常。
解决方式是本地调试时临时把后台前端请求地址切换为本地后端地址。
前端页面虽然是在本地打开，但它请求的后端不一定是本地，需要确认环境配置实际指向哪里。

10. 后台菜单不显示问题

后台新增菜单后，重启服务并重新登录，菜单仍然没有显示。
这个问题排查过程比较典型，最后发现不是单一原因，而是多个因素叠加：

  • 原因一：菜单版本号不一致。系统获取菜单时会按照当前公司版本过滤权限，新增菜单的版本号一开始和其他系统管理菜单不一致，因此被过滤掉了。解决方式是把新菜单的版本号调整成和同级菜单一致。
  • 原因二：当前角色没有分配新菜单权限。虽然测试账号所属角色名称叫“超级管理员”，但系统权限逻辑里它并不是自动拥有全部权限的账号。新菜单添加到权限表后，还需要分配给当前角色，分配后这个角色才有权限看到菜单。
  • 原因三：后台服务连接的是远程数据库。本地后台前端虽然请求的是本地后端，但本地后台后端服务连接的仍然是远程数据库和远程缓存，而新增菜单和反馈数据是在本地数据库中添加的。解决方式是把本地后台服务的数据库和缓存配置也临时切换到本地。
    最终前端、本地后台服务、本地数据库统一后，菜单正常显示。

11. 本地调试环境要统一

这次调试中最容易混淆的点，就是前端、后端、数据库不一定连的是同一套环境。
最终需要保证：

  • 小程序前端请求本地小程序后端；
  • 小程序后端连接本地数据库；
  • 后台管理端前端请求本地后台后端；
  • 后台后端也连接本地数据库；
  • 本地缓存也保持一致。
    只有这样，小程序提交的数据才能在后台管理端看到。
    如果其中任何一环连到了远程环境，就可能出现“明明本地数据库有数据，但页面看不到”的情况。

12. 图标资源处理问题

小程序意见反馈入口需要新增图标。
过程中曾经使用过 SVG 转 PNG，但项目中直接放 SVG 文件可能导致构建或运行异常。最终删除了不再使用的 SVG 文件，只保留 PNG 图片资源。
同时对 PNG 图标做了几次视觉调整，让它和原有“我的”页面图标风格更接近，包括大小、圆形背景、内部图形居中和线条风格。
这个问题说明：新增资源时不只是能显示就行，还要考虑项目对资源格式的兼容性和整体视觉一致性。

13. 提交前需要注意的配置回退

本地调试过程中，为了让功能跑通，临时修改了多个环境配置。
这些配置包括：

  • 小程序请求地址；
  • 后台管理端请求地址；
  • 后端数据库地址；
  • 后端缓存地址；
  • 小程序 AppID；
  • 本地开发工具配置。
    这些都只是本地调试用，不能直接提交到代码仓库。
    提交前必须检查并恢复为公司原来的远程配置，否则可能影响同事环境或正式测试环境。

总结

这次意见反馈功能从小程序提交到后台查看，完整打通了前端、后端、数据库和后台管理端。

2026.6.9 对管理后台的问题处理

1. 反馈类型枚举值不统一

开发过程中，前后端对“反馈类型”的取值没有一开始就统一。最初设计是：1 = 问题反馈、2 = bug反馈；后来为了和项目里其他下拉选项的习惯保持一致，调整为：0 = 问题反馈、1 = bug反馈。但小程序端没有及时重新构建，导致仍然提交旧值，结果后台显示时出现“问题反馈被显示成 bug反馈”和“bug反馈显示成 -”。

最终解决：小程序提交值改为 0 / 1；后台管理端展示逻辑同步改为 0 / 1；数据库字段注释和变更 SQL 同步更新；修正本地测试数据中的旧枚举值。

2. 后台删除按钮不显示

后台页面已经加了删除按钮和权限判断 showBtnFn('system:feedback:remove')，数据库中也插入了权限记录，但页面仍然看不到删除按钮。

  • 原因一：权限类型字段配置错误。权限表中用于区分“菜单”和“按钮”的字段配置错误，原本删除权限被配置成菜单类型 menu_sort = 0，正确应该是按钮类型 menu_sort = 1。修正后，删除权限才能作为按钮权限返回给前端。
  • 原因二：权限缓存未刷新。项目后端会把用户权限缓存到 Redis 中，缓存有效期较长，即使数据库权限已经修正，前端重新登录时仍可能拿到旧的权限树。
  • 最终解决：修正删除权限的类型字段；清理 Redis 中对应的权限缓存；重新登录后台。

3. 测试图片上传到了远程文件服务

小程序端上传图片时，请求的是本地后端接口，但后端文件上传配置指向的是远程测试文件服务，因此测试过程中上传的图片实际被上传到了远程对象存储。

最终处理：查询本地反馈记录中保存的图片 URL；删除本地测试反馈记录；使用项目已有删除接口，只删除本次测试上传的两张图片；通过访问图片 URL 确认返回 404 / NoSuchKey，确保图片已删除。

这个问题提醒：本地后端不一定代表所有资源都在本地，文件服务、Redis、数据库等配置需要分别确认。

4. 查看详情样式不统一

一开始后台详情页使用普通文本展示，后来改成 disabled 输入框，但样式仍然和项目已有页面不一致。对比项目中已有的“组织结构查看页”后，发现公司已有查看页通常是：复用表单结构，使用 el-form，使用 disabled 的 el-input / el-select，保持和新增/编辑页面相同的表单样式，只是不允许操作。

最终调整：反馈类型使用 disabled 下拉框；普通信息使用 disabled 输入框；反馈内容使用 disabled 多行文本框；图片使用预览组件；详情抽屉布局参考已有查看页。

5. 提交前恢复本地配置

开发过程中，为了本地调试修改过一些配置，例如小程序 AppID、前端接口地址、后端数据库地址、Redis 地址、微信相关配置。这些配置只适合本地开发，不能提交到远程仓库。最终提交前统一检查三个项目的 Git 状态，只保留功能代码，回退所有本地环境配置。

6. 最终提交情况

本次功能涉及三个项目：

  • 小程序端： 新增意见反馈入口；新增反馈类型选择页；新增反馈填写页；支持图片上传、预览、删除；提交反馈到后端接口。
  • 后台管理端： 新增意见反馈管理页面；支持列表展示、筛选、分页；支持查看详情；支持图片预览；支持删除反馈；接入按钮权限控制。
  • 后端： 新增反馈表实体；新增 Mapper / Service / ServiceImpl；新增小程序端保存接口；新增后台管理端分页查询接口；新增后台管理端删除接口；新增数据库变更 SQL。

最终三个项目均提交到远程开发分支，没有影响主分支。

总结： 这次问题主要集中在前后端枚举值不统一、权限字段配置错误、权限缓存没有刷新、本地环境和远程文件服务混用、查看样式需要和已有页面保持一致，以及提交前需要严格排查本地配置。
