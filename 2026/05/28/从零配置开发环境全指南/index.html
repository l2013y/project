从零配置开发环境全指南

换电脑或重装系统后最头疼的事，莫过于从头配一遍开发环境。本文不仅整理了 Git、JDK（1.8/17/21）、Maven、SDKMAN、Node.js（16/24）、Hexo 博客搭建、SSH 密钥配置的安装和配置步骤，还会说明每个工具是干什么的、为什么需要它——知道”为什么装”比”怎么装”更重要。

Git

Git 是什么？ Git 是一款分布式版本控制系统。通俗地说，它能记录你每一次代码修改的内容、时间、作者，让你可以随时回退到历史版本、对比代码差异、多人协作开发时合并各自的改动而不会互相覆盖。几乎所有软件开发团队都在用 Git 做代码管理。

为什么需要它？ 写代码难免改坏或误删，Git 就像”存档系统”，随时可以读档。多人合作一个项目时，Git 能自动合并各人的修改，冲突时提示手动解决。GitHub 和 GitLab 则是基于 Git 的代码托管平台，相当于把 Git 仓库放到云端。

Windows 和 macOS 的安装方式略有不同。

Windows 安装

去官网 https://git-scm.com/downloads/win 下载安装包，选择对应系统位数（64 位或 32 位）。安装过程基本一路 Next，但有几个选项建议留意：

  • 选择默认编辑器：建议选 Nano 或 Notepad++，不要选 Vim（除非你熟悉）
  • 调整 PATH 环境：选 “Git from the command line and also from 3rd-party software”（第二个选项）
  • 行尾换行符：选 “Checkout Windows-style, commit Unix-style line endings”（第一个选项）
  • 终端模拟器：选 “Use MinTTY”

安装完成后新开终端，验证一下：

git --version
git version 2.47.0.windows.1

macOS 安装

推荐用 Homebrew：brew install git。如果没有 Homebrew，先去 https://brew.sh 按指引安装。也可以用 Xcode Command Line Tools 自带的 Git：在终端输入 git --version，系统会自动提示安装。

基础配置

装完 Git 后先配置用户名和邮箱，这是每次 commit 的身份标识：

git config --global user.name "你的用户名"

git config --global user.email "你的邮箱@example.com"

顺便把换行符策略也设一下，避免 Windows 和 macOS 协作时出问题：

git config --global core.autocrlf input （macOS）

git config --global core.autocrlf true （Windows）

查看所有配置：git config --global --list。

JDK（1.8、17、21）

JDK 是什么？ JDK（Java Development Kit）是 Java 开发工具包，包含两个核心部分：javac（编译器，把 .java 源码编译成 .class 字节码）和 java（运行时，执行编译好的字节码）。简单说：写 Java 程序靠它编译，跑 Java 程序靠它运行。

JRE 和 JDK 的关系？ JRE（Java Runtime Environment）只含运行时，只能运行 Java 程序不能编译。JDK 包含 JRE，所以装了 JDK 就不用再装 JRE。

为什么列了 8、17、21 三个版本？ 这三个都是 LTS（长期支持）版本，企业和开源项目广泛使用。不同项目可能锁定在不同版本——老项目用 JDK 8，中间升级用 JDK 17，新项目用 JDK 21，所以经常需要同时安装并在它们之间切换。

去哪里下载

推荐两个渠道：

  • Oracle 官网：https://www.oracle.com/java/technologies/downloads/ — 需要注册账号
  • Adoptium（Eclipse Temurin）：https://adoptium.net/download/ — 开源免费，无需注册，推荐用这个

下载时注意选 .msi（Windows）或 .pkg（macOS）安装包，一路 Next 即可。安装路径建议记一下，后面配环境变量要用。

Windows 环境变量配置

  1. 右键”此电脑” → 属性 → 高级系统设置 → 环境变量
  2. 在系统变量中新增 JAVA_HOME，值填 JDK 安装路径，比如 C:\Program Files\Eclipse Adoptium\jdk-17.0.9.9-hotspot\
  3. 编辑系统变量的 Path，新增一条 %JAVA_HOME%\bin
  4. 确定保存后新开终端验证：

java -version
openjdk version “17.0.9” 2023-10-17

多版本切换（Windows）

同时装了 JDK 8、17、21 的话，最简单的切换方式是改 JAVA_HOME，指向不同版本的安装路径。也可以把三个版本的环境变量都配好（JAVA_HOME8、JAVA_HOME17、JAVA_HOME21），让 JAVA_HOME 引用其中一个：在环境变量中设为 %JAVA_HOME17%，切换时改这一个值就行。

macOS 多版本管理

macOS 上 JDK 装完后路径在 /Library/Java/JavaVirtualMachines/。多版本切换可以用 SDKMAN（见下一节），或者在 ~/.zshrc 中加一行：

export JAVA_HOME=$(/usr/libexec/java_home -v 17)

把 17 换成对应版本号即可。

SDKMAN

SDKMAN 是什么？ SDKMAN（Software Development Kit Manager）是一款 SDK 版本管理工具。它的作用是让你用一行命令安装、卸载、切换不同版本的 JDK、Maven、Gradle 等开发工具，而不用手动下载安装包、配环境变量。它相当于 Java 生态的”nvm”。

为什么需要它？ 如果只装一个 JDK 版本，手动配环境变量就够了。但当你在两个项目间切换——一个用 JDK 8，一个用 JDK 21——手动改环境变量会很繁琐。SDKMAN 一条 sdk use 命令就能在当前终端临时切换，不影响其他终端。

注意：SDKMAN 原生支持 macOS / Linux / WSL，Windows 下需要配合 Git Bash 使用。

安装 SDKMAN

打开终端执行：

curl -s "https://get.sdkman.io" | bash

装完后执行 source "$HOME/.sdkman/bin/sdkman-init.sh" 使其生效，或关掉终端重开。验证：

sdk version
SDKMAN 5.18.2

常用命令

sdk list java — 列出可安装的 JDK 版本，结果很多可以过滤：sdk list java | grep "17\."

sdk install java 17.0.9-tem — 安装指定版本（版本号从 list 结果里复制）

sdk install java 21.0.1-tem — 再装一个 21

sdk use java 17.0.9-tem — 当前终端临时切换到 JDK 17

sdk default java 21.0.1-tem — 设为默认版本，新终端自动使用

sdk current java — 查看当前使用的版本

sdk install maven 3.9.6 — 用 SDKMAN 装 Maven 也一样方便

用 SDKMAN 管理 JDK 的好处是不用手动配环境变量，切版本一行命令搞定。

Maven

Maven 是什么？ Maven 是 Java 生态中最主流的两大构建工具之一（另一个是 Gradle）。它主要做三件事：

  1. 依赖管理：你在配置文件 pom.xml 里声明项目需要哪些第三方库（比如 Spring Boot、MyBatis），Maven 自动从中央仓库下载这些 jar 包以及它们依赖的其他 jar 包，不用手动去网上找和复制。
  2. 构建生命周期：Maven 定义了标准化的构建流程——编译（compile）、测试（test）、打包（package）、部署（deploy），每一步都有对应的命令。
  3. 项目结构约定：Maven 项目有统一的目录结构（src/main/java 放源码，src/test/java 放测试），开发者接手任何 Maven 项目都能立刻知道代码在哪。

和 Gradle 的区别？ Gradle 使用 Groovy/Kotlin DSL 配置，构建脚本更灵活简洁，性能也更快，Android 开发首选 Gradle。Maven 使用 XML 配置，更啰嗦但更标准化，传统企业级 Java 项目大多用 Maven。

Windows 安装

  1. 去官网 https://maven.apache.org/download.cgi 下载 Binary zip 包
  2. 解压到目标目录，比如 C:\tools\apache-maven-3.9.6
  3. 配置环境变量：新增 MAVEN_HOME 指向解压目录，Path 里加 %MAVEN_HOME%\bin
  4. 验证：mvn -version

Node.js（16、24）

Node.js 是什么？ Node.js 是一个 JavaScript 运行时环境，它让你可以在浏览器之外（比如服务器、命令行、桌面应用）运行 JavaScript 代码。之前 JavaScript 只能跑在浏览器里，Node.js 把它带到了后端。

npm 又是什么？ 装好 Node.js 后会自动附带 npm（Node Package Manager），它是 Node.js 的包管理器，相当于 Java 世界的 Maven——在 package.json 里声明依赖，npm 自动下载。常用的 yarn 和 pnpm 也是包管理器，功能类似但各有优势（pnpm 更快更省磁盘）。

为什么列了 16 和 24？ 不同项目对 Node 版本要求不同，老项目可能锁在 Node 16（LTS），新项目用 Node 24。多个项目并行开发时就需要在版本间切换。

安装 nvm

Windows 用户下载 nvm-windows：https://github.com/coreybutler/nvm-windows/releases，选 nvm-setup.exe 安装。

macOS / Linux 用户执行：

curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash

装完重启终端验证：nvm version。

常用 nvm 命令

nvm list available — 查看可安装版本（Windows），macOS 用 nvm ls-remote

nvm install 16.20.2 — 安装 Node 16

nvm install 24.0.0 — 安装 Node 24

nvm use 24.0.0 — 切换到 Node 24

nvm list — 查看已安装的版本

nvm alias default 24.0.0 — 设默认版本

切换 Node 版本后，之前全局安装的 npm 包不会自动迁移。可以在安装新版本时用 --reinstall-packages-from 迁过来：

nvm install 24.0.0 --reinstall-packages-from=16.20.2

npm 配置

npm config set registry https://registry.npmmirror.com — 设淘宝镜像，加速下载

npm install -g yarn — 全局安装 yarn

npm install -g pnpm — 全局安装 pnpm

验证安装：

node -v
v24.0.0

SSH 密钥配置

SSH 密钥是什么？ SSH（Secure Shell）密钥是一对加密文件——私钥（你保管，不要泄露）和公钥（放到服务器上）。连接时，服务器用公钥加密一段随机数据，只有持有对应私钥的你能解密，从而验证身份。

为什么需要它？ 没有配 SSH 密钥时，每次 git push、git clone 都要输入 GitHub / GitLab 的账号密码，频繁操作很烦。配好密钥后，Git 客户端自动用私钥完成认证，无需每次输密码，而且比密码更安全（密钥长度 256 位以上，暴力破解几乎不可能）。

生成 SSH 密钥

打开终端，执行以下命令生成 ed25519 类型的密钥（更安全、更快）：

ssh-keygen -t ed25519 -C "你的邮箱@example.com"

如果系统较老不支持 ed25519，用 RSA：ssh-keygen -t rsa -b 4096 -C "你的邮箱@example.com"。

执行过程中会提示选择保存路径和设置密码短语（passphrase）：

  • 保存路径直接回车，使用默认的 ~/.ssh/id_ed25519
  • 密码短语可设可不设，设了每次使用密钥都要输；不设直接回车跳过

添加到 ssh-agent（macOS / Linux）

eval "$(ssh-agent -s)" — 启动 ssh-agent

ssh-add ~/.ssh/id_ed25519 — 将密钥添加到 agent

Windows 用户装 Git 时会自带 OpenSSH，Git Bash 里直接使用即可。

配置 GitHub

  1. 复制公钥内容：cat ~/.ssh/id_ed25519.pub，把输出全部复制
  2. 登录 GitHub → Settings → SSH and GPG keys → New SSH key
  3. Title 随便填（比如”我的笔记本”），Key 粘贴刚复制的公钥，点 Add SSH key
  4. 验证连接：ssh -T git@github.com

Hi 你的用户名! You’ve successfully authenticated, but GitHub does not provide shell access.

看到这条就说明成功了。

配置 GitLab

步骤基本一样：

  1. 登录 GitLab → Preferences → SSH Keys
  2. Key 粘贴公钥，Title 随便填，Expiration date 可留空表示永不过期
  3. 验证连接：ssh -T git@gitlab.com

Welcome to GitLab, @你的用户名!

Hexo 博客从零搭建

Hexo 是什么？ Hexo 是一个静态博客生成框架。你只需要用 Markdown 写文章，然后执行一条命令，Hexo 就会自动把文章转换为一个个美观的 HTML 静态页面，可以直接部署到 GitHub Pages 等免费托管平台。

和 WordPress 等动态博客的区别？ WordPress 需要服务器、数据库，功能强大但维护成本高。Hexo 生成的博客是纯 HTML/CSS/JS 静态文件，不需要服务器和数据库，部署到 GitHub Pages 完全免费，加载速度也更快。代价是没有后台管理界面，写文章、改配置都在本地编辑器完成然后推送到仓库。

为什么选它？ 免费、快、Markdown 写作体验好、社区主题多，适合程序员和技术写作。

准备工作

开始前确认 Node.js 和 Git 已装好（node -v、git --version），然后在 GitHub 注册账号并创建一个仓库，仓库名必须是 你的用户名.github.io（比如 zhangsan.github.io），设为 Public。

安装 Hexo

npm install -g hexo-cli

hexo init my-blog

cd my-blog

npm install

初始化完成后，项目目录结构如下：

  • _config.yml — 站点配置文件（最重要，大部分配置都在这里改）
  • source/_posts/ — 存放 Markdown 文章
  • source/_drafts/ — 存放草稿
  • themes/ — 主题目录
  • public/ — 生成的静态文件（运行 hexo generate 后产生）

运行 hexo server，浏览器打开 http://localhost:4000 就能看到默认页面。

写文章

hexo new "文章标题" — 在 source/_posts/ 下生成 .md 文件

hexo new draft "草稿标题" — 创建草稿，存在 source/_drafts/ 下

hexo publish "草稿标题" — 把草稿发布为正式文章

文章文件头部的 --- 之间可配置标签和分类：tags: [Hexo, 博客]、categories: 教程。正文直接写 Markdown，随时 hexo server 预览。

安装主题

默认主题较朴素，推荐换一个。以 Next 主题为例：

git clone https://github.com/next-theme/hexo-theme-next themes/next

下载后改 _config.yml 里的 theme: next，重启 hexo server 看效果。每个主题有自己独立的配置文件，具体参考主题文档。

项目迁移

换电脑后，只需要备份这几个文件/目录就能完整恢复博客：

  • _config.yml、package.json
  • source/（所有文章和图片）
  • themes/（如有自定义主题配置）
  • scaffolds/（如有自定义模板）

不需要备份 node_modules/、public/、.deploy_git/，这些可重新生成。建议把整个 Hexo 项目推到一个私有仓库，换电脑时直接 clone → npm install → hexo server 就能跑。

总结

以上是从零配置开发环境的完整流程，按顺序走一遍大约半个小时。几个建议：

  • 包管理器优先：macOS 用 Homebrew、SDKMAN、nvm，能省去大量手动配环境变量的时间
  • 多版本管理：SDKMAN 管 Java，nvm 管 Node，多项目并行开发时切换自如
  • 国内镜像：Maven 和 npm 都配好国内镜像，下载体验天差地别
  • SSH 密钥：配一次管所有仓库，别每次都输密码
  • 备份思维：把关键配置文件（.zshrc、settings.xml、_config.yml 等）存到云端，换电脑时省得从头来
