知识点

return 是关键字，不是函数。关键字后面跟反引号，就只是普通模板字符串，没有”标签模板”这个魔法机制。而 console.log 是函数，函数名后面直接跟反引号（没有括号），就触发标签模板了。

示例数据：

情况一： console.log(…) —— 反引号里的模板字符串作为普通参数传入，输出的是拼接好的字符串。

商品名称：香蕉,价格：3,目前无库存 …

情况二： console.log…`` —— 函数名后没有括号，直接跟反引号，这是标签模板调用。此时 console.log 拿到的第一个参数是字符串片段数组，${...} 里的值作为后续参数依次传入。

一句话记：括号 = 普通调用，参数是那个字符串本身；无括号 + 反引号 = 标签模板，参数是”拆开的字符串数组 + 各插值”。

find vs filter

  • find 找”第一个满足条件的元素”，返回元素本身；filter 找”所有”，返回数组。
  • filter 找不到 → 空数组 []；find 找不到 → undefined。
  • 数组用复数命名，回调参数用单数命名（社区惯例）。

比较运算符

  • = 是赋值，返回被赋的值；if (x = 1) 永远进 if 分支。
  • == 是宽松相等，会做类型转换，规则反直觉。
  • === 是严格相等，类型不同直接 false。永远用 ===。
  • x == true 是冗余：如果 x 本身是布尔值，直接 x 就行。

可选链 ?.

  • x?.y：如果 x 是 null 或 undefined，整个表达式返回 undefined，不再往右读；否则等同 x.y。
  • 只处理 null / undefined，其他假值（0、''、false）不管。
  • 杀手锏是嵌套访问：response?.data?.user?.profile?.name。
  • 明知左边一定有值时不要用，会掩盖 bug。

空值合并 ??

  • x ?? y：如果 x 是 null 或 undefined，返回 y；否则返回 x。
  • 跟 ?. 是黄金搭档：user?.age ?? 0。
  • ?? vs ||：|| 判所有假值（0、''、false 都替换），?? 只判 null / undefined。当 0 / 空串 / false 是合法值时必须用 ??。
    核心规则：
    • ?. 防的是”读属性时对象是 null/undefined”
    • ?? 防的是”值已经安全拿到了，但它是 null/undefined，我要个默认”
    • 两个一起用：?. 先保命，?? 再兜底

try/catch

  • 作用：接住抛出的错误，防止程序中断。JS 默认出错会让报错行之后的代码全部不执行。
  • 能”判断”就不要”抓错”：
    • 能提前检查的（find 找不到、字段缺失）→ 用 if / ?. / ??
    • 无法提前检查的（网络请求、JSON 解析、外部输入）→ 用 try/catch
  • ?. 防御一个值，try/catch 防御一段代码。

this

this 不是定义时决定的，是调用时决定的。规则：看函数被调用那一行，. 点号前面是谁，this 就是谁。函数一旦被”取”出来（存变量、传参、作为回调），this 就跟原对象断联。

根本原因：函数是一种值，可以传递存储，本身不携带”我属于谁”的信息。

对象 vs 数组

  • {} 是对象（键值映射，无序），[] 是数组（有序索引）。
  • obj[key] = 值 同时管”加字段”和”改字段”——不存在就创建，存在就覆盖。
  • obj.key 找字面量字段 key；obj[key] 先算变量再找字段。动态字段名只能用方括号。
  • 这是 JS 分组、映射、动态字段所有代码的基础。

if 后不加 else

表示”准备 + 行动”两步走，不是二选一。

异步与 Event Loop

  • 判断异步看”操作类型”（setTimeout / fetch / 事件），不看有没有 async 关键字。
  • 铁律：同步代码不跑完，任何异步都不能插队——哪怕 setTimeout(fn, 0) 也一样。
  • new Promise(fn) 里的 fn 是同步立刻执行的，只有内部包裹的 setTimeout / fetch 才是异步（反直觉）。

Promise 三态

  • 三种状态：pending / fulfilled / rejected，切换后永远锁死。
  • resolve(x) 是”完成的那一瞬间”通知机制，不是”开始”——写错位置会让异步彻底失效。
  • resolve(x) ≠ console.log(x)：前者通知 Promise，能让 .then 收到；后者只是打印。不调 resolve，.then 永远不触发。
  • resolve 和 return 相似但不同：resolve 不中断函数、能被多个 .then 接收。

链式调用

  • .then 里 return 什么，下一个 .then 的参数就是什么。
  • 想让下一步”再等一段时间” → return new Promise(...)（.then 里根本没有 resolve）。
  • .then(成功函数, 失败函数)——两个位置都是函数，不是字符串。
  • 链条末端要用 console.log，不要 return（return 出去没人接就丢了）。

函数传参：fn vs fn()

  • fn 传的是函数本身（未来执行）；fn() 是立刻执行，把返回值传过去。
  • 异步 API 需要”引用”，不是”调用结果”。传错就变成”现在就跑，把返回值给未来”，异步失效。

async / await

  • async 函数永远返回 Promise（不管 return 什么）。await 必须在 async 函数里（top-level await 是 ES2022 模块里的特例）。
  • await 和 .then 二选一，别叠用——await 已经拿了值，.then 就多余。
  • async/await 不是 Promise 的”简化版”，而是”消费 Promise 的一种写法”。Promise.all / new Promise / .race 该用还得用。
  • 同步操作 await 是白写——await 1 + 1 没意义。
  • 不加 await 会拿到 Promise { <pending> }——代码不等，Promise 是”待兑现的支票”。
  • async 函数被 await 切成一段段的珠子：段内同步、段间让权。异步状态不能”取消”，只能”走完”。

fetch 的坑

  • fetch 只在网络层挂了才 reject（域名解析失败、断网、CORS）。HTTP 404 / 500 不算失败——要手动 if (!response.ok) throw 才能让 catch 抓到。
  • fetch 要两次 await：第一次拿盒子（Response 对象），第二次拆盒子（.json() 解析成 JS 对象）。
  • .json() 是”把服务器发来的 JSON 字符串解析成 JS 对象”，方向要搞清楚。反过来发数据用 JSON.stringify()。
  • Response 是浏览器内置构造函数，跟自己拿到的响应对象 response 不是一回事——大小写陷阱。

DOM 渲染姿势

  • innerHTML = HTML 字符串——不是塞对象，是塞字符串，浏览器自己解析成真元素。直接塞对象会调 toString() 变成 "[object Object]"。
  • map + join('') 是列表渲染标准姿势：map 每个元素变 HTML 片段，join 把数组拼成一大段字符串。不加 join 会出现逗号——数组转字符串默认用逗号拼。
  • CSS 行内样式必须是 属性名:值 格式：style="color:red" 而不是 style="red"。
  • DOM 操作固定姿势：先拿容器（const el = document.getElementById(...) 提到最前），再操作容器。别依赖”浏览器把 id 自动挂到 window”这种黑魔法。

三态渲染（列表页四态）

  • 四态：loading / success / error / empty——真项目每一态都要处理。
  • loading 必须放在 await 之前——await 之后再塞 loading，用户等的那 1 秒是空白。
  • 判空要在渲染之前 + return 提前退出——顺序反了 return 就白写。
  • 空数组 map / filter / join 都不报错，反而必须主动判空 + 显示”暂无数据”，否则用户分不清”没数据”和”页面挂了”。

JS 数据类型全景

补充：typeof null === 'object'（历史 bug）→ 判 null 用 x === null；typeof [] === 'object' → 判数组用 Array.isArray(x)。6 个假值：false / 0 / '' / null / undefined / NaN。原始类型按值传递，引用类型按”指向”传递——const b = a 如果 a 是数组，改 b 会牵连 a。

箭头函数

  • 三步简化：去 function、加 =>、能省的都省。
  • {} 和 return 捆绑：要么都写要么都省。n => { n * n } 会返回 undefined。
  • 返回对象必须用 () 包裹：id => ({ id, name })，否则 { 被当成函数体。
  • 参数括号：只有 1 个光秃秃参数才能省；0 个 / 多个 / 带默认值 / 解构 / rest 都必须写。
  • 键名与变量名同名时可简写：{ id: id } → { id }。

事件系统

  • e.target（被点的，跨监听器动态变）vs e.currentTarget（绑监听的，同一监听器固定）。
  • e.preventDefault() 阻止默认行为；e.stopPropagation() 阻止冒泡。
  • 冒泡三阶段：捕获（外→内）→ 目标 → 冒泡（内→外），默认监听器都在冒泡阶段响应。
  • 事件委托靠冒泡实现：父容器 1 个监听 + e.target 判断 + classList.contains 或 dataset。
  • 委托内层嵌套元素：e.target.closest('.delBtn') 或 CSS pointer-events: none。
  • dataset 读出来永远是字符串，要数字必须 Number(e.target.dataset.id)。
  • tagName 返回大写（'BUTTON'），判断要写大写。
  • oninput 每敲一字都触发（看过程，实时字数统计、实时搜索）；onchange 失焦且值变化才触发（看结果盖章，表单最终校验、下拉选择）。只失焦不改内容 change 不触发。
  • .onclick（点击）/ .oninput（输入变化）/ .onchange（值改变且失焦）/ .onmouseover（鼠标移入）/ .onkeydown（按键按下）/ .onsubmit（表单提交）——命名规律是 on + 事件动词，想绑什么事件先猜英文动词加 on，scroll / focus / blur / load 全对。
  • el.onXxx = fn vs el.addEventListener('xxx', fn) 三个真实差异：前者只能绑一个（后绑覆盖前绑）、事件名带 on 前缀、解绑 el.onclick = null；后者能绑无限个、事件名不带 on 前缀（写 'onclick' 无效）、解绑要 removeEventListener('click', 那个函数引用)。现代主流用 addEventListener——React / Vue / 所有框架内部都用它。
  • removeEventListener 靠函数引用找监听器——匿名函数解不了绑（每次写出来的匿名函数在内存里都是不同的函数）。想以后解绑就得先存变量。不解绑的后果是内存泄漏：组件走了监听器还在，堆积多了页面越用越卡。这也是 React useEffect return cleanup 存在的原因之一。

变量声明 const / let / var

  • const 锁”变量指向”，不锁”数据内容”——const arr = []; arr.push(4) 合法，arr = [新数组] 不合法。
  • 判断依据：= 左边只有变量名 → 报错；带 .属性 或 [索引] → 合法。
  • 默认写 const，需要重新赋值才改 let；var 不用。
  • const / let 没有变量提升——存在暂时性死区，必须先定义再使用。写在下面用、写在上面死。
  • 链式赋值 a = b.属性 = 函数 是陷阱：= 从右往左结算，且赋值语句本身也是表达式，b = 值 的值就是”值”本身。const btn = document.getElementById('id').onclick = fn 会让 btn 装的是 fn 不是按钮。别一行同时做”绑事件”和”存变量”两件事。
  • JS 不检查拼写：变量名对 JS 只是一串字符，suqareOutput 拼错两遍照样运行。能跑 ≠ 写对了。养成用 F2 重命名的习惯，别手打两遍希望没错。这也是 TypeScript 存在的核心动机——把”变量名/类型错误”从”运行时崩溃”提前到”写代码时红波浪线”。

数据 vs 视图

  • DOM 是数据的一次投影：改 DOM 不改数据，后续依赖数据的逻辑（渲染、判空、统计）全错。
  • .remove() 只删 DOM；filter / splice 才是删数据。
  • 刷新丢数据 ≠ 数据没删干净，那是”持久化”（localStorage）的另一个话题。
  • 现代前端主流是”数据驱动”：只改数据，视图自动重画（React / Vue / 小程序）。

表单读值 input.value

  • 可读可写：input.value 能读也能写，input.value = '' 可以清空输入框。只属于 <input> / <textarea> / <select> 这类”用户能输入”的元素。
  • 永远是字符串：<input type="number"> 读出来也是 "22" 不是 22——参与数学运算前必须 Number() 强转。
  • 读取时机：必须在”用户已经输完”的信号里读（onclick / oninput / onchange 内部），文件最外层读到的是空串。
  • 边界消毒原则：从 <input> 读出来的 .value 是”脏数据”，一读进来就 Number() 转掉，别让它污染后续代码——这个原则以后接收 API 参数、读数据库、处理用户输入都适用。

表单提交与多控件

  • <form> 里的 <button> 默认 type="submit"——放 form 里点了就会提交（页面刷新），想只当普通按钮必须显式写 type="button"。
  • 表单提交监听 form 的 submit，不是 button 的 click——submit 同时覆盖鼠标点击 + 回车键；click 只覆盖鼠标。浏览器有条隐藏规则：焦点在 form 里的 input 时，回车会自动模拟点击第一个 type="submit" 按钮。
  • e.preventDefault() 阻止事件默认行为——表单场景就是阻止刷新页面。
  • .value 永远是字符串（type="number" 也一样）；判空用 value.trim() === '' 或 .length === 0，不要 === 0——字符串跟数字 0 永远不 ===。
  • <select> 读被选中项的中文：select.selectedOptions[0].textContent（复数别漏 s——即使单选模式也叫 selectedOptions，因为 select 本身支持多选）。别用 for 循环遍历。
  • checkbox 读勾选状态用 .checked（布尔值），不是 .value。.value 在 checkbox 上永远是 "on"（历史遗留，别用）。
  • 一组 DOM 元素的处理 = 数组家族的思路：querySelectorAll 拿一组 → Array.from 转真数组 → filter / map / join 想筛就筛想变形就变形。filter / map 不只是”处理数据的”，是”处理任何可遍历东西的”，DOM 组也算。
  • checkbox 组读被勾选项的完整链条：Array.from(querySelectorAll('input[name="xxx"]')).filter(el => el.checked).map(el => el.value).join('、')。关键是手里一路握着元素本身，直到最后一步才 map 成字符串——一开始就 map(el => el.checked) 把元素压扁成布尔值，后面就退不回去了。

DOM 元素查找 querySelector 家族

方法 能按什么找 返回 场景
getElementById('x') 只能 id 一个元素或 null 独一无二的元素
querySelector('...') 任何 CSS 选择器 第一个匹配或 null 单个但没 id
querySelectorAll('...') 任何 CSS 选择器 NodeList（类数组） 一组元素
  • CSS 选择器常用三种：#id / .class / [属性="值"]（如 input[name="hobby"]）；可组合，input[name="regGender"]:checked = 所有 name="regGender" 且被选中的 input。
  • radio 组读值一句话：document.querySelector('input[name="xxx"]:checked')——选中就返回那个元素、未选返回 null。
  • querySelectorAll 返回的是 NodeList，不是真数组，要 Array.from(...) 转真数组才能用完整的 map / filter / reduce。
  • 一组元素永远用 querySelectorAll——不要给一组元素都加 id（id 必须唯一），也不要给外层容器加 id 再钻进去遍历（那是在用蛮力重新发明 querySelector）。
  • CSS 选择器只描述”选什么元素”，判断（.length === 0 / > 3）必须写在字符串外面：querySelectorAll('input[name="hobby"]:checked===0') 里的 ===0 会被当成选择器语法崩掉。
  • **NodeList 直接跟数字比永远 false**——querySelectorAll(...) === 0 是拿对象跟数字比，勾 0 个也过、勾 4 个也过。要先 .length 拿数字：querySelectorAll('input[name="hobby"]:checked').length——直接数勾了几个，不用 Array.from + map 绕一圈。

变量命名带类型提示

  • 复数 → 数组：hobbies / checkedElements。
  • Count 后缀 → 数字：checkedCount。
  • Text 后缀 → 字符串：checkedText。
  • Info / Data 后缀 → 对象：userInfo。
  • 参数名说”它是什么”（item / order），变量名说”它装的是什么形状的东西”。看名字就能猜出下一步能对它调什么方法。

类型流意识

  • 链式方法每一步在心里”贴新标签”：querySelector(...) → DOM 或 null；querySelectorAll(...) → NodeList；.value → 字符串（永远是！）；.value.length → 数字；.checked → 布尔；Array.from(NodeList) → 真数组；.filter → 数组（元素类型不变）；.map → 数组（元素可能变类型）；.join(分隔符) → 字符串。
  • Array.from 只对”复数”用：NodeList 转真数组合法；对单个 DOM 或 null 用会崩或得到空数组。
  • 模板字符串里 ${} 必须走到最终类型：${input} 得到 "[object HTMLInputElement]"，要写 ${input.value.trim()}。写链式方法之前先把每一步”贴什么标签”想清楚，比写完再调试快很多。
  • 表单校验通用骨架：清空 → 变量准备 → 一串 if...return 校验 → 成功动作放在 if 结构外（不是又一个分支，是所有校验没触发的自然结果）。e.preventDefault() 阻止浏览器默认提交，return 阻止你自己写的提交代码——两个搭配才是完全停止。

逻辑运算符 || / &&

  • || 一真则真、&& 一假则假——不只是判断规则，还是”短路”机制：见到定型的值立刻停、右边不执行。
  • 中文”和”很暧昧，有时候是”且”有时候是”或”。写代码前先问：”两个条件全都成立才动，还是只要一个成立就动？”全都要 &&、任一 ||。想说”长度 < 2 或 > 10 就报错”写成 &&，就变成”既 < 2 又 > 10”，一个数字永远不可能同时成立，条件永远 false。
  • 由短路衍生的三种简写：a || 默认值（兜底）/ 对象 && 对象.属性(安全访问) / 条件 && 动作()（简版 if）。

报错解读：Cannot read properties of XXX (reading 'YYY')

  • 翻译：你在 XXX（null / undefined）上读 .YYY。
  • 排查套路：找 reading 'YYY' → 定位代码里读 .YYY 那行 → 那行点号左边就是空的那货。
  • 前端 90% 的运行时报错都长这样——搞懂这一句能省一大堆调试时间。

编码心法：早退 & 校验前置

  • 提前返回（early return）：函数开头先用 if...return 把不合规情况挡掉，主逻辑无缩进写在最后。收益：省性能（未通过就不做无用功）+ 主逻辑显眼 + 加校验只是加一行不动主体。反例是 if / else if / else 层层嵌套，主逻辑埋在洋葱最里层。
  • 校验前置、读值后置：先挡门再读值——挡住的情况不做无用读值。
  • 一个业务 = 一个事件回调，别在回调里再嵌 addEventListener。addEventListener 是叠加绑定（不像 onclick= 会覆盖），嵌一层每触发一次就多绑一个监听器,第二次点触发两个、第三次三个，渐进式变卡——典型的事件泄漏。
  • else if 长条 = 学生代码：每个分支都 return 时，else 是废话——独立 if 即可。更工程化的做法：const errors = [] 收集所有错误，一次 push 一次显示 → 用户能一次看清所有毛病。判断代码”专不专业”的一个信号：想到”用户填错怎么办”是学生思维；想到”用户会填错几个、怎么让他一次改完”是工程师思维。
  • return 管函数不管块：return 无视 if / for / while 这些”块”，抬头找最近的 function 或 () =>，找到就把整个函数从这里截断。所以校验段结尾 if (errors.length > 0) { ...; return } 拦的不是错误，是错误发生时不该跑的正常流程——快照、按钮变化、setTimeout、卡片渲染，一整段全冻结。
  • 重复代码结构一致 = 抽成数据：一串 if 每条都是”如果[某条件]成立，就报[某消息]”——变化不多 → 塌缩成 const rules = [{ check, message }, ...] + forEach 遍历。antd / Element / React Hook Form / Zod 这些校验库底层都是这个心智，只是套了更多壳子。
  • 存值 vs 存动作（惰性求值）：check: 表达式 在对象创建那一刻就被算成固定的 true / false——之后用户改输入框追不上。check: () => 表达式 存的不是”立刻算出的结果”，而是”待命的一段代码”，只有 rule.check()（加括号调用）时才执行。变量存值、函数存动作——这也是 React useEffect 依赖数组”stale closure”坑的同一原理：箭头函数被创建那一刻，外部变量就被冻在闭包里。
  • 词法作用域：函数抬头找变量只顺着”自己写在哪”往外找，不看”调用时刻有没有同名变量”。const pfGender = ... 声明在 submit 函数体内就只活在那个函数里，顶层的箭头函数看不见——把箭头函数改成自己读 DOM（() => !document.querySelector(...)）就能跨房间取物。
  • forEach 里操作形参不操作数组名：rules.forEach(rule => rule.check)——rules 是整个数组（袋子），rule 是循环里当前那颗。写成 rules.check 会拿到 undefined；对象没有 .push()，rules.push(...) 直接崩。数组用复数、元素用单数不是玄学。

异步交互三个”绑定时机”辨析

  • 按钮.onclick = fn → 存起来，下次点才跑。
  • 事件回调里直接写 按钮.textContent = '...' → 这次点击立刻执行。
  • 想响应”本次点击”，用不着再绑 onclick——已经在这次点击的回调里了，顺着往下写就行。
  • 渲染代码放哪里由”什么时候显示”决定：立刻显示 → 事件回调同步位置；N 毫秒后 → setTimeout 回调里面；页面加载就显示 → addEventListener 外面的顶层。位置一错，要么秒执行、要么根本没执行、要么变量没定义就报错。

拍快照心法（异步表单必备）

  • 异步的两条铁律：什么时候跑（Event Loop）+ 跑的时候读的是哪个时刻的数据（快照）。前者是排队规则，后者是数据保鲜——两个都得懂。
  • 判断依据：变量装的是”字符串本身”（快照）还是”通往 DOM 的引用”（不是快照）。
  • .map(el => el.value) 出来的数组是快照（值已复制）；pfName 这种 DOM 引用不是——pfName.value 每次都是”现问”，用户 1.5 秒内改了输入框，读到的就是改后的。
  • 想拿”点击那一刻的数据” → 同步阶段先 const snapshot = { name: pfName.value, age: pfAge.value } 打包好，异步回调里只用 snapshot.name。
  • 场景：发消息、点赞、下单、上传——凡是”点击后有异步延迟 + 用户可继续操作 UI”的都得拍。
  • 底层原理：原始类型（字符串/数字）存进变量 = 复制一份，冻住；引用类型（DOM/对象/数组）存进变量 = 存的是指针，.value 每次都现问。

运算符对字符串的行为

  • +：只要有一边是字符串就走拼接 → "22" + 1 = "221"。
  • - * / %：自动强转数字再算 → "22" * 2 = 44。
  • 写 price * count 巧合能对，但一旦某处出现 +（比如加运费）整条数据链就崩——+ 是叛徒。别靠 * 碰运气，源头 Number() 消毒。

函数是一种”值”（两句总纲）

  • const 名字 = function(){} = 给函数起名，自己以后调用（第①类：定义函数）。
  • xxx.事件名 = function(){} = 把函数挂到对象属性上，等系统在特定时机自动调用（第②类：绑定事件监听器）。
  • 右边都是函数（因为函数是”值”，和数字/字符串同级），但左边完全是两件事：起名备用 vs 挂钩子。
  • 匿名函数：filter / map / onclick 回调不需要名字，因为没人需要”通过名字找到它”，直接作为参数传出去就行——箭头函数就是为这种”一次性回调”设计的简写。

return 判断口诀

  • return 是函数把值”交出去”的唯一通道。
  • 有人接才 return，没人接就别写。
  • filter / map / find / sort / reduce / .then 的回调 → 必须 return（这些方法在等你的返回值）。
  • onclick / oninput / setTimeout 里的 fn / 末端 .then 打印 → 不要 return（浏览器 / 系统不看返回值）。
  • 写 return 之前先问自己一句：”谁在接？”

函数抽象与重构

  • 抽函数三步走：先建空壳 function xxx() {}，再搬代码进壳里，最后原地叫一声 xxx()。只搬代码不建壳，代码会直接在顶层执行，还读不到未定义的变量。
  • 函数间沟通只靠三样：参数进、return 出、词法作用域可见的外部变量。函数抬头找变量按”写在哪”找，不按”何时调用”找——变量声明在函数体内，函数外的箭头函数看不见。
  • return 只跳出自己所在的函数，跳不出外层回调。校验函数里 return 拦不住外层 submit 后续代码——把结果用返回值告诉外面（if (!validate()) return）。这跟”守卫子句”是同一个 pattern：!validate()将上面返回的布尔值取反，if () return 表示if语句正确的话就停止之后的代码”。
  • 布尔方向别反：true = 通过、false = 没通过（按英文单词字面意思用）。有错 return false，没错 return true。
  • 形参 vs 实参：定义时括号里是形参（占位标签），调用时是实参（实际的货）。形参已经是函数私有变量，函数体里再 const 同名 会报错。抽通用函数要”形参写了就要用”——函数体里出现具体名字（loginError、pfError）就说明没抽干净。
  • 布尔 vs 数组返回值：布尔只告诉”过没过”，数组能告诉”过没过 + 具体错在哪 + 有几条”，信息量差一个数量级。工程上优先返回数据，判断留给调用者——同一个 validate 能给不同表单复用，因为它只算错误、不管怎么显示。
  • 单一职责：一个函数只做一件事。validate 只算错误，不管怎么显示——显示是调用者的自由。想通用就要把”显示错误”从函数里搬出来，放回调用者身边。
  • 函数命名约定：validate（校验）/ render（渲染）/ handleXxx（事件处理）/ getXxx / fetchXxx（获取）/ isXxx（布尔状态）。名字对代码运行毫无影响，但决定别人（和三个月后的自己）能不能秒读。
  • 对象字面量键值用冒号：{ check: () => ..., message: '...' }——键值之间是冒号，等号是变量赋值。

2026.7.6 filter 与 find 的十一个坑

背景：JS 复习到数组方法家族，filter 是第一站，find 是第二站。这是我从 1.1 一路踩到 2.3，十一道题里踩过的所有坑的复盘。

坑 1：把 filter 当 forEach 用（1.2）

第一版对象数组筛选，回调里只有 console.log，跑起来 inventory 是空数组。filter 的回调必须 return 一个布尔值——true 保留，false 丢弃。没写 return，JS 默认 return undefined，是假值，于是每个元素都被判为”不留”。

console.log 是打印给人看的，return 是交回给调用者用的。filter 不看你打印了啥，只看你 return 了啥。

坑 2：return 一个对象当筛选条件（1.2）

修正版写了 return item，结果 inventory 变成了全部 5 个商品，一个都没筛掉。在 JS 里对象是真值（truthy），return item 相当于永远 return true。filter 想要的不是”任何值”，是布尔表达式，要写 return item.stock > 0。

坑 3：参数作用域（1.2）

参数只在函数大括号里活着，出了 } 就消失了。要打印的是 inventory，不是 item——inventory 才是 filter 交还给我的东西。

坑 4：filter 不检查语义（1.2）

return item.price > 0——我想筛”有库存的”，但写成了 price。语法完全正确，5 个商品价格全大于 0，inventory 又变成了 5 个。filter 只干”根据布尔值筛选”这一件事，不检查你筛的字段对不对，判断得对不对是写代码的人的责任。写 filter 前先在脑子里问一句”我到底要留什么样的元素？”再动手。

坑 5：参数命名 vs 变量命名（1.3）

组合条件”筛有库存且价格 < 10 的商品”，我把参数名改成了 cheapItem，觉得”更符合语义”。但 filter 会把每个元素都传进回调，包括草莓（15 元）、西瓜（20 元）这种贵的——这时候把它叫 cheapItem 就是说谎。

  • 参数名描述”每次输入的一个元素”，应该叫 item / product，语义稳定不随需求变
  • 变量名描述”筛完的结果”，可以叫 cheapInventory，跟随业务语义

如果需求改成”筛贵的”，参数叫通用名 item 只需改变量名和条件（2 处）；参数叫 cheapItem 则要改变量名、参数名（两处出现）、条件（5 处）。恰恰因为需求会变，参数才更要用不会变的通用名。

坑 6：x == true 是冗余的（1.4）

return user.age >= 18 && user.active == true——user.active 本身就是布尔值，再拿它跟 true 比一次是多此一举。另外 JS 里 == 会做类型转换（0 == false 是 true），容易造 bug。正确写法：return user.age >= 18 && user.active。永远用 ===，别用 ==。

坑 7：user.name = '小丽' 用了单等号（find 2.1）

想比较用户名，写成了赋值。find 停在第一个元素，返回被篡改过的对象。写比较用 ===，别用 =。敲完等号先问自己”要改还是要比”。

坑 8：users.name 而不是 user.name（find 2.1 & 2.3）

数组名（复数）和参数名（单数）只差一个 s，手抖写成了 users.name，得到 undefined，条件永远为 false。数组用复数、元素用单数是社区惯例——看到 users.name 立刻反应”这是整个数组的字段，不是当前元素”。

坑 9：把变量命名为 undefined（find 2.2）

想表示”这次没找到”，写了 const undefined = ...，报 Identifier 'undefined' has already been declared。undefined 是 JS 内置标识符，不能覆盖。变量名描述”想装什么”，不描述”这次装了啥”。用 foundUser / targetUser。

坑 10：find 找不到时访问属性直接爆炸（find 2.2）

targetUser.age 在 targetUser 是 undefined 时报 Cannot read properties of undefined。find 找不到返回 undefined，不是空数组。访问属性前必须防御——用 ?. 或 if 判断。

坑 11：结果对但逻辑错（find 2.3）

回调忘写 return，找”张三”依然输出 undefined。结果看起来对，其实是”找不到”和”没 return”两个 bug 撞出的巧合。换成找”小丽”就露馅。写完代码不要只跑”你想让它对”的输入，也要跑”应该找到”的输入验证。结果对 ≠ 代码对。

2026.7.10 跨过 reduce 这道坎

背景：数组家族最后一站。filter / find / sort / map 的回调都是”处理一个元素”，reduce 完全是另一个思路——它是”折叠器”，把一整个数组”折”成任何东西（数字、字符串、对象、数组都行）。推进到 4.4 都很顺，4.5 分组统计卡了很久，最后靠退回去练基础语法打通的。

核心心智模型

雪球 / 擂台赛

reduce 每一轮有两个角色：

  • acc（累加器 / 擂主 / 雪球）：跨越多轮的”记忆”
  • current（当前元素 / 挑战者 / 新雪花）：本轮上场的那个

每一轮 return 什么，下一轮 acc 就是什么。函数里的局部变量每轮都会重置——只有 acc 能跨轮存东西。

找最大值不用两两都比

一开始困惑：”5 和 3 没直接比过，怎么能确定最大值？”后来想通了——只要每个挑战者都跟”当前擂主”打过一次，最后的擂主一定是最强的。3 打不过 20，那 3 有没有比 5 大就已经不重要了。n 个元素只需要 n 次比较。

reduce 什么都不”整理”

它就一件事：拿着 acc 和 current，调用回调，把 return 值当新 acc。你想要什么形状的结果，就在回调里手动把 acc 塑造成那个形状。它对结果类型毫无感知。

六大真实场景

reduce 不是”只能求和”——能描述”两个东西怎么合成一个”，就能用 reduce 折整个数组：

  1. 累加：购物车总价、订单金额
  2. 找极值：最贵的商品、最新的评论（对象数组用 reduce 比 Math.max 好用）
  3. 数组 → 对象（分组）：按状态分类的订单、按首字母分组的联系人 ⭐
  4. 数组 → 字符串：复杂拼接（简单拼接用 join）
  5. 数组去重：对象按 id 去重
  6. 多指标一次统计：数据看板里一次算出总数、总额、平均值

踩坑清单

坑 1：不写初始值遇上对象数组 → "[object Object]200"

不写初始值时 acc 会”借”第一个元素当起点。数字数组还好，对象数组会把 acc 变成整个对象，然后”对象 + 数字”被强转成怪字符串。

坑 2：空数组 + 不写初始值 → 直接抛 TypeError

程序崩溃。真实场景里空数组是常态（新用户、无搜索结果、空列表），只要写 reduce 就写第二个参数。

铁律：初始值不仅是起点，也是”告诉 reduce 你最终想要什么类型”。要数字写 0、要对象写 {}、要数组写 []。

坑 3：以为局部变量能存跨轮的值

每次回调都是全新的函数调用，局部变量从头初始化。想要”记住”任何东西，必须存进 acc。

坑 4：acc + 0 加了个寂寞

条件不满足时写 return acc + 0——加零没意义，代码里看到多余操作反而让读者多想。能省则省，直接 return acc。

坑 5：命名把用途当成本体

写过 carts.reduce((acc, sum) => ...)——把 current 参数命名为 sum。但 sum 让人以为是”总和”，实际它是”当前商品”，读代码时会短暂困惑。

方法 进（原数组） 出（返回值） 会改原数组吗？ 一句话记忆
map 任何数组 数组（长度不变） 不改 一对一变形
filter 任何数组 数组（长度 ≤ 原来） 不改 筛选，元素不变
find 任何数组 单个元素 或 undefined 不改 找第一个
reduce 任何数组 任何类型（看初始值） 不改 折叠成一个
sort 任何数组 原数组本身（改过的） 改！ 原地排序
forEach 任何数组 undefined（没返回值） 不改 只是走一遍
join 任何数组 字符串 不改 拼接成字符串
push / splice 任何数组 数字 / 数组 改！ 原地操作

两条规律值得记住：返回新数组的都不改原数组（map / filter），原地操作的返回值往往不是你想要的那个东西（sort 返回自身、push 返回新长度）。forEach 是异类——它什么都不返回，纯粹走一遍。

2026.7.13 第一次踏进异步世界：从 setTimeout 到 Promise

背景：之前学的所有代码——变量、数组家族、reduce、DOM——全都是同步。今天第一次踏进异步的世界。从 setTimeout 开始理解异步的本质，一路走到回调地狱、Promise 三态、链式调用。

异步的本质

某些操作会”挂起等未来时机”，主线不停下来等它，继续跑；等时机到了，被挂起的代码才回来执行。核心铁律是：同步代码不跑完，任何异步都不能插队——哪怕 setTimeout(fn, 0) 也一样。这条规则叫 Event Loop（事件循环）。

回调地狱

要写”顺序依赖”的异步流程（下单 → 支付 → 发货 → 收货），只能用 setTimeout 层层嵌套——代码往右下”倒”成一个金字塔。真实业务里嵌套 8-10 层的历史遗留代码非常多，读起来眼睛跟不上层级、加逻辑要挪一堆括号、每层都要单独处理错误。

Promise 三态与链式调用

Promise 是一个”承诺对象”——代表一个未来会有结果的异步操作。用 resolve(x) 通知成功、reject(y) 通知失败，用 .then(fn) 预先注册”结果到手要干的事”。

.then 里 return 什么，下一个 .then 的参数就是什么；想让下一步再等一段时间，就 return 一个新的 Promise——下一个 .then 会等新 Promise 完成再执行。金字塔式的嵌套被拍平成从上往下的链条。

一个顿悟：Promise 优化的不是行数，是复杂度和可维护性。步骤少 + 无依赖时看不出优势；步骤多 + 有数据传递 + 要错误处理时完胜回调。而且性能几乎无损耗——底层还是回调。

踩坑清单

坑 1：以为 resolve = “启动异步操作”

真相相反——resolve 是”完成的那一瞬间”才调，不是”开始”。如果理解反了，会把 resolve 写在最外层立刻执行，Promise 秒变 fulfilled，异步彻底失效。

坑 2：console.log(结果) 和 resolve(结果) 是两回事

console.log 只是”打印给人看”，外部拿不到；resolve 才是”通知 Promise，让 .then 收得到”。不调 resolve 只 console.log，.then 永远不触发——这个坑第一次写 Promise 时几乎必踩。

坑 3：在 .then 里直接用 resolve

resolve 只存在于 new Promise(function(resolve, reject){...}) 的函数体内部。.then 的回调里没有 resolve，直接用会报 resolve is not defined。想在 .then 里”再等一段时间”，正确姿势是 return 一个新的 new Promise。

坑 4：链条最后一步 return 了但没人接

.then 里最后一步 return 一个值，没有下一个 .then 来打印——那个值就丢了。终点站直接 console.log，不 return。

坑 5：以为 .then 里有一个位置是”填字符串”

.then(成功函数, 失败函数)——两个位置都是函数。里面 console.log('拿到:', res) 那种”字符串 + 变量”是 console.log 自己的参数，跟 .then 结构无关。

坑 6：把 fn 传成 fn()

fn = 传函数本身（未来执行），fn() = 立刻执行，把返回值传过去。异步 API 需要”引用”，不是”调用结果”。传错就变成”现在就跑，把返回值给未来”，异步失效。

2026.7.15 箭头函数 + 事件系统入门

背景：JS 复习进入语法简化和交互两个模块。上半场把 function 一路简化成箭头函数，把三条简写规则和三种参数场景摸清；下半场从事件对象、冒泡、委托、dataset 一路串到”数据 vs 视图”的老问题，顺手把 const / let 到底锁什么、JS 数据类型全景又深化了一层。

上半场：箭头函数

从 function(x){ return x*2 } 到 x => x*2 的三步简化：去掉 function、加 =>、能省的都省。三条简写规则：

  • 单参数省括号：(x) => ... 简写成 x => ...。
  • **单行 return 省 {} 和 return**：x => { return x*2 } 简写成 x => x*2。这里 {} 和 return 是捆绑的，要么都写要么都省——写了 n => { n * n } 会返回 undefined，因为大括号被当成函数体，里面没 return。
  • 返回对象要用 () 包裹：id => ({ id, name })，否则 { 会被当成函数体的开始。

参数括号规则：只有”1 个光秃秃参数”才能省。0 个 / 多个 / 带默认值 / 解构 / rest 都必须写括号——只要参数写法稍微复杂一点，规则就退回”必须有括号”。

下半场：事件系统入门

事件对象 e： e.target 是”被点的那个元素”，跨监听器动态变；e.currentTarget 是”绑监听的那个元素”，在同一监听器里固定。e.preventDefault() 阻止默认行为（比如表单提交），e.stopPropagation() 阻止事件继续冒泡。

冒泡三阶段： 捕获（外→内）→ 目标 → 冒泡（内→外）。默认监听器都在冒泡阶段响应——事件从最内层往外一层层传，路上绑了监听的都会触发一次。

事件委托： 只在父容器上绑一个监听器，靠 e.target + classList.contains() 判断具体点的是谁。省事件监听、动态元素也能自动生效。委托的实现原理就是冒泡——内层元素被点，事件冒到父容器时被统一处理。

内层嵌套的坑： 按钮里嵌了 <span>，点到 span 时 e.target 是 span 而不是按钮，classList.contains('delBtn') 判不到。修法两选一：e.target.closest('.delBtn') 往上找最近的匹配元素，或者给 span 加 pointer-events: none 让它对点击透明。

dataset 属性： HTML 里写 data-id="1"，JS 里 element.dataset.id 读取。值永远是字符串——要数字必须 Number(e.target.dataset.id)。另外 e.target.tagName 返回大写（'BUTTON'），判断要写大写。

穿插补的：变量声明 + 数据类型 + 数据 vs 视图

let vs const： let 是能重新赋值的变量声明；const **只锁”变量指向”，不锁”数据内容”**。判断依据是等号左边：只有变量名 → 重新赋值 → 报错；带 .属性 或 [索引] → 改内容 → 合法。所以 const arr = [] 之后 arr.push(4) 完全合法，只是 arr = [新数组] 不合法。默认写 const，需要重新赋值才改 let，var 不用。

JS 数据类型全景： 7 种原始类型（string / number / boolean / undefined / null / bigint / symbol）+ 引用类型（Object / Array / Function 等）。原始类型按值传递，引用类型按”指向”传递——const b = a 如果 a 是数组，改 b 会牵连 a。

数据 vs 视图（再深化一层）： li.remove() 只删 DOM 不删数据，todoArray.length 还是原来的数字。状态与视图一旦脱钩，后续依赖数据的逻辑（渲染、判空、统计）就会连锁出错。现代前端主流是”数据驱动”——只改数据，视图自动重画（React / Vue / 小程序）。刷新丢数据是另一个话题（localStorage），跟”数据没删干净”不是一回事。

踩坑清单

坑 1：HTML 属性写错位置

class 挂到闭合标签上了，实际按钮身上没有这个 class，事件委托里 classList.contains('delBtn') 永远为假。属性只能写在开标签里。

坑 2：变量名大小写不一致

声明的是 todoArrays（大 A），赋值时手抖写成 todoarrays（小 a）——JS 不报错，而是创建了个新变量，原数组永远没被改过。JS 严格区分大小写，todoArrays 和 todoarrays 是两个完全不同的变量。

坑 3：以为 const 就是”不可变”

以为 const 声明的东西一动不了，结果 arr.push(4) / user.name = '小红' 都合法。const 锁的是”变量指向哪个仓库”，不锁”仓库里放什么”。想真正锁住内容要用 Object.freeze()。

坑 4：箭头函数返回对象忘加括号

id => { id, name } 会被 JS 当成函数体大括号，里面 id, name 是逗号表达式，最后没 return，返回 undefined。返回对象必须写 id => ({ id, name })。

坑 5：{} 和 return 只省了一半

n => { n * n } 返回 undefined——大括号被当成函数体，里面缺 return。简写规则里 {} 和 return 是捆绑的，要么都写（n => { return n * n }）要么都省（n => n * n）。

坑 6：DOM 删除 ≠ 数据删除

li.remove() 后 todoArray.length 还是原来的数字。DOM 只是数据的一次投影，改投影不影响原本。要真删数据得靠 filter / splice。

修复：让箭头函数自己读 DOM，不依赖函数内的变量。() => !document.querySelector('input[name="pfGender"]:checked')——document 是全局能访问的入口，箭头函数在顶层也能用。

坑 5 · 把 JS 判断塞进 CSS 选择器字符串

修 hobby 那两条时写成 document.querySelectorAll('input[name="pfHobby"]:checked===0')——===0 被当成了选择器语法的一部分。CSS 选择器只描述”选什么元素”，判断（.length === 0、> 3）必须写在字符串外面。

坑 6 · 救出来的时候把 .length 也丢了

把判断从字符串里救出来时写成 document.querySelectorAll(...) === 0——丢了 .length。querySelectorAll 返回的是 NodeList（一个对象），对象跟数字直接比大小永远是 false，所以勾 0 个能过、勾 4 个也能过。”有几个”不是选择器的属性，是选完之后 NodeList 的 .length。

一个心得

这一天最大的顿悟是**”stale closure” 的原理跟 React useEffect 完全一样**——箭头函数被创建那一刻，外部变量就被冻在闭包里。React 依赖数组坑就是这个东西，只是被 hooks 的壳子包着不明显。今天在原生 JS 里提前撞了一次，以后学 React 就是”哦，这个见过”。

另一个体感是**”代码变数据”**：一串结构一致的重复代码 → 抽成 [{ check, message }, ...] + forEach，这是校验库、路由表、菜单配置、表格列定义所有”配置化”代码的地基。学会这个之后再看框架源码，看到的不是魔法而是同一个模式在不同层次上反复出现。

2026.7.24 把校验段抽成通用函数

背景：接着 7.10 的规则表版本继续往前推。这一次不是”多加一条规则”，是把整段校验逻辑抽成独立函数，再把它变得能给任何表单复用。四小步走完，函数抽象这件事第一次真正落地。

第一步 · 抽函数（Extract Function）

把 submit 回调里的一整段校验逻辑——forEach 遍历 rules、errors 收集、innerHTML 显示、return 拦截——打包成独立函数 validate，放在顶层。submit 回调里只保留一句 if (!validate()) return。改造完后 submit 回调瘦了一半，一眼看出主流程是”校验 → 拍快照 → 延迟 → 渲染卡片”。

第二步 · 参数化

页面新增登录表单，规则和错误显示区都不一样。这时候发现 validate 跟 profile 那套 DOM 绑死了——函数里写死的 rules、pfError 改成形参 rules、errorEl。同一个 validate，两处调用传不同实参：validate(profileRules, pfError) 和 validate(loginRules, loginError)。

第三步 · 单一职责

写着写着发现 validate 同时干了两件事——校验 + 显示错误。把”显示”那段从函数里搬出来放回调用者身边。validate 只留下”跑规则收集错误”，参数只剩 rules，返回值从 true / false 改成 errors 数组。空数组代表通过、非空代表有错——信息量比布尔值大一个数量级。

第四步 · 复用验证

再加一个留言表单，从 DOM → msgRules → submit 回调整套流程走一遍。整个过程没有改 validate 函数一行——证明抽象是真的通用。

踩坑清单

坑 1 · 抽函数只搬代码没建函数壳

把 validate 里的代码剪出来直接丢在文件顶层——没有 function xxx() {} 包裹。刷新页面立刻执行，还读到未定义的变量。修法：抽函数分两步，先建空壳 function validate() {}，再把代码搬进壳里。

坑 2 · 函数看不见变量（词法作用域）

把 validate 放顶层、rules 和 errors 留在 submit 回调里 → 函数抬头找不到这两个变量，报 errors is not defined。词法作用域按”写在哪”找，不按”何时调用”找。修法：rules 搬到顶层（跟 DOM 常量平级），errors 留在函数里（每次调用新建）。

坑 3 · return 跳错层

validate 里的 return 只能跳出 validate 自己，跳不出 submit 回调。校验没通过还是继续拍快照 + setTimeout。修法：validate 用返回值告诉外面结果（true / false），submit 里用 if (!validate()) return 拦下。

坑 4 · 布尔方向反了

if (errors.length > 0) { ... return true }——有错的时候告诉外面”通过了”。修法：true = 通过、false = 没通过（按英文单词字面意思用）。有错 return false，没错 return true。

坑 5 · if (!validate()) return 卡在”false 是关是开”

一直纠结布尔值到底是”开”还是”关”。修法：读大白话——“如果校验没通过就 return”。!validate() = “没通过”，if () return = “就滚”。跟 7.10 的守卫子句是同一个 pattern。

坑 6 · 把成功分支当成错误分支

以为 submit 里 if (!validate()) return 后面的 setTimeout + 卡片渲染是”错误提示逻辑”。其实错误提示在 validate 函数内部就贴到页面上了，submit 后面的代码全是”通过后才跑的成功分支”。搞清楚这一点后第一步才算真通关。

坑 7 · 参数当变量重新声明

function validate(rules, errorEl) { const rules = [...] }——参数已经是函数私有变量，再 const 声明会报错。参数就是函数的私有变量，函数体里直接用，不用再声明。

坑 8 · 抽通用函数抽了个”半吊子”

函数签名写了 (rules, errorEl)，函数体里还用具体名字 loginError.innerHTML——等于没参数化。修法：形参写了就要用，函数体里出现具体名字（loginError、pfError）就说明没抽干净。

坑 9 · 把对象当函数调用

rules.forEach(rule => { if (rule()) ... })——rule 是对象 { check, message }，不是函数。报错 rule is not a function。修法：rule.check()——调用对象里的 check 属性。

坑 10 · 对象字面量里用了 = 不是 :

{ check: () => ..., message = '...' }——message 后面写成等号 → 语法错误。对象里键值之间用冒号，等号是变量赋值。

坑 11 · 边界条件符号选错

“5-50 字以内”写成 length <= 5 || length >= 50 → 正好 5 字和 50 字都被判为错。题目说”允许的边界”用 < / >，不用 <= / >=。

坑 12 · event 类型选错

表单校验监听 submit 而不是 click——click 抓不到回车键提交，preventDefault() 也只对 submit 事件有效。这个坑 7.17 踩过一次，今天又踩了。

一个心得

这一天最大的收获是**”通用性验证”这件事：一个函数抽得干不干净，不是自己看着觉得”应该没问题”，而是换个场景直接拿来用，一行不改就跑通**。第三个表单从 DOM 到规则表到 submit 回调重写了一遍，validate 那 5 行代码完全没动——那一刻才真正相信这个抽象是有效的。

另一个体感是”抽函数三步走”和”函数间沟通三通道”这两条口诀。以前看别人的代码觉得函数拆得挺自然，自己一动手就发现每一步都有坑：不建壳直接搬、参数留一半具体、单一职责糊在一起。把这些坑踩一遍，下次抽函数的时候手感就完全不一样了。
